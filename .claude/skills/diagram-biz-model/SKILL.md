# Diagram Biz Model

ビジネスモデルを**自分（ビジネスモデル・サービスデザインの知識・経験が低いSE/PL）**が、お金・情報・データの流れとステークホルダー関係をビジュアルで構造的に理解できるHTMLに変換する。

## 想定読者

- **自分自身**（ビジネスモデル・サービスデザイン・ビジネスデザインの知識・経験が低い）
- ゴール: 図解を見て「お金・情報・データの流れが**構造として頭に入り**、人に説明できる状態」になること

---

## ワークフロー

### Phase 0: 対象の把握

入力を整理する:

- **対象**: 企業名・サービス名・ビジネスモデル名
- **関心の核心**: 何を知りたいか（「なぜ安い？」「誰がお金を出している？」「データをどう使っている？」）
- **疑問点**: わかっていること・わからないことを分ける
- **文脈**: なぜ今これを知りたいのか（検討中の事業・参考にしたい事例など）

### Phase 1: 徹底調査（サブエージェント）

**exploreサブエージェント**を起動して、対象ビジネスモデルを徹底調査する。

```
Task({
  subagent_type: "explore",
  description: "ビジネスモデルの徹底調査",
  prompt: `
    以下のビジネスモデルを徹底的に調査してください：
    【対象】{企業名・サービス名}
    【関心】{何を知りたいか}
    
    調査項目：
    1. 主要ステークホルダー（誰が関わっているか）
    2. お金の流れ（誰が誰に何を払っているか）
    3. 情報・データの流れ（誰が誰に何を提供しているか）
    4. 価値交換の核心（なぜこのモデルが成立するか）
    5. コスト構造の特徴（なぜ安い・高い・持続可能か）
    6. 類似ビジネスモデルの名前（プラットフォーム型・サブスク型など）
    
    結果は箇条書きで整理して返してください。
  `
})
```

### Phase 2: ステークホルダーの整理

登場人物と役割を整理する:

| ステークホルダー | 役割 | 提供するもの | 受け取るもの |
|----------------|------|------------|------------|
| （例）消費者 | | | |
| （例）企業 | | | |
| （例）パートナー | | | |

### Phase 3: フロー設計

3種類のフローを明確化する:

| フロー種別 | 方向 | 内容 |
|----------|------|------|
| お金 | A → B | 何のための支払いか |
| 情報 | A → B | どんな情報か |
| データ | A → B | どんなデータか・どう使われるか |

### Phase 4: HTML生成

ステークホルダーとフローを視覚化したHTMLを生成する。

**詳細** → [references/html-structure.md](references/html-structure.md)

### Phase 5: 批判的レビュー（サブエージェント）

**ビジネスモデルに興味はあるが知識・経験が低い人の視点**で、忖度なくレビューする。

```
Task({
  subagent_type: "generalPurpose",
  description: "ビジネスモデル図解の批判的レビュー",
  readonly: true,
  prompt: `
    あなたは「ITエンジニア歴8年だが、ビジネスモデル・経営・マーケティングの知識はほぼゼロ」の人物です。
    ビジネス書を読んだことはほぼなく、BtoBとBtoCの違いはわかるが、プラットフォームやサブスクの仕組みは曖昧。
    
    以下のHTML図解を読んで、**忖度なく**批判してください：
    
    【HTMLファイルパス】{path}
    
    ## レビュー観点
    
    1. **わからないビジネス用語**: 説明なしに出てくる用語はどれか
    2. **フローの追いやすさ**: お金・情報・データの流れが矢印を追えば理解できるか
    3. **「なぜ」が腑に落ちるか**: ビジネスモデルが成立する理由が直感的にわかるか
    4. **ステークホルダーの多さ**: 登場人物が多すぎて混乱しないか
    5. **説明できるか**: 読み終えて「30秒で人に説明できる気がするか」
    
    ## 出力形式
    
    ### 致命的な問題（これがあると理解できない）
    - ...
    
    ### 改善すべき点（直せばもっと良くなる）
    - ...
    
    ### 良かった点（このまま残すべき）
    - ...
    
    ### 総評
    5段階評価: ⭐⭐⭐☆☆（3/5）のように
    一言: 「○○だから△△は理解できるが〜はわかりにくい」のように
  `
})
```

### Phase 6: ブラッシュアップ

Phase 5のレビュー結果を踏まえてHTMLを修正する。

**優先度**:
1. 致命的な問題 → 必ず修正
2. 改善すべき点 → 可能な限り対応
3. 良かった点 → 維持

### Phase 7: デプロイ

```bash
# Windows環境ではGit BashまたはWSLで実行する
DEPLOY_DIR=$(mktemp -d)
# index.html を $DEPLOY_DIR に書き込む
cd "$DEPLOY_DIR" && surge . --domain {対象名をkebab-caseにしたもの}-biz.surge.sh
```
