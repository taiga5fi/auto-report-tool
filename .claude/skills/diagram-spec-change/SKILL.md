# Diagram Spec Change

システム仕様変更を**電力系統・担当システム仕様の知識が低いエンジニア**でも、変更前後の状態と影響を一人で判断できるHTMLに変換する。

## 想定読者

- プログラム開発のプロだが**電力系統・業務背景の知識が低い**エンジニア（約20名）
- ゴール: 図解を見て「変更前後の状態・影響を**自分で判断できる**状態」になること

---

## ワークフロー

### Phase 0: 変更依頼の把握

入力を整理する。不明な場合はユーザーに質問する:

- **変更対象**: 何の実装か（機能名・処理名・モジュール名）
- **変更内容**: どう変えたいか（削除・修正・追加）
- **問いかけ**: 「なぜその変更が必要と思っているか」（背景・根拠）
- **関連仕様**: どの仕様項目に該当するか（わかれば）

### Phase 1: 業務背景・仕様の整理

エンジニアが理解できるよう、業務ルールと仕様を平易な言葉で整理する:

- この処理が「何のために存在するか」（業務的な意義）
- 電力系統・業務固有の用語をエンジニア向けの言葉に置き換え
- 仕様書の該当箇所の引用（ある場合）

**翻訳の方針**: 「〇〇（業務用語）とは、△△（エンジニアが知っている概念）と同じ。エンジニアに例えると〜」の形で説明する。

### Phase 2: 変更前後の整理

変更前と変更後を以下の観点で比較する:

| 観点 | 変更前 | 変更後 |
|------|--------|--------|
| システムの状態 | | |
| この処理が何をしているか | | |
| 処理が発動する条件 | | |
| 処理が持つ効果 | | |

### Phase 3: 影響分析

実装を変更した場合のリスクを明確化する:

- **業務への影響**: 電力系統のルール・契約・規制に違反しないか
- **他処理への連鎖**: 他のモジュール・処理が依存していないか
- **特定条件下の挙動**: エッジケースで何が変わるか
- **影響なしの根拠**（削除可能な場合）: なぜ影響がないと言えるか

### Phase 4: HTML生成

変更前後の比較・影響を視覚化したHTMLを生成する。

**詳細** → [references/html-structure.md](references/html-structure.md)

### Phase 5: 批判的レビュー（サブエージェント）

**電力系統・業務知識がゼロのエンジニア視点**で、忖度なくレビューする。

```
Task({
  subagent_type: "generalPurpose",
  description: "仕様変更図解の批判的レビュー",
  readonly: true,
  prompt: `
    あなたは「プログラム開発は10年以上のベテランだが、電力系統の知識はゼロ」のエンジニアです。
    システム仕様書を読んだことはなく、業務ルールは知らない。コードは読めるが業務的な「なぜ」は知らない。
    
    以下のHTML図解を読んで、**忖度なく**批判してください：
    
    【HTMLファイルパス】{path}
    
    ## レビュー観点
    
    1. **わからない業務用語**: 説明が不十分で意味が掴めない言葉はどれか
    2. **変更前後の比較がわかるか**: 何が変わるのかが一目でわかるか
    3. **影響が判断できるか**: この変更をしていいか、自分で判断できる気がするか
    4. **情報の多さ**: 判断に不要な情報で混乱する箇所はないか
    5. **結論が出るか**: 図解を読んで「OKかNGか」の意見が言える状態になるか
    
    ## 出力形式
    
    ### 致命的な問題（これがあると判断できない）
    - ...
    
    ### 改善すべき点（直せばもっと良くなる）
    - ...
    
    ### 良かった点（このまま残すべき）
    - ...
    
    ### 総評
    5段階評価: ⭐⭐⭐☆☆（3/5）のように
    一言: 「○○だから△△な判断は難しい」のように
  `
})
```

### Phase 6: ブラッシュアップ

Phase 5のレビュー結果を踏まえてHTMLを修正する。

**優先度**:
1. 致命的な問題 → 必ず修正
2. 改善すべき点 → 可能な限り対応
3. 良かった点 → 維持

### Phase 7: デプロイ

```bash
# 1. 一時ディレクトリ作成
DEPLOY_DIR=$(mktemp -d)

# 2. HTMLファイル作成
# index.html を $DEPLOY_DIR に書き込む

# 3. surge.shにデプロイ
cd "$DEPLOY_DIR" && surge . --domain {変更名をkebab-caseにしたもの}-spec.surge.sh
```

※ Windows環境ではGit BashまたはWSLで実行する。
