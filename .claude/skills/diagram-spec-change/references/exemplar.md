## 成功する図解の構造

```
1. ヘッダー
   └─ 変更対象名 + 一行説明 + 「要確認 / 変更OK」バッジ

2. この処理は何か（業務背景）
   └─ 業務用語の解説ボックス（初出時）
   └─ 「何のために存在するか」を平易に説明

3. 仕様書の該当箇所
   └─ 仕様文の引用
   └─ 仕様とコードの対応

4. 変更前後の比較
   └─ 変更前カード（赤）/ 変更後カード（緑）の並列表示
   └─ 「変化のポイント」強調ボックス

5. 影響分析
   └─ 影響先 × 影響内容 × 発生条件 × レベルのテーブル

6. 判断サポート
   └─ 変更OK / NG / 要確認の根拠を箇条書き
```

## 業務用語の翻訳パターン

### 成功する翻訳の形

```
「〇〇（業務用語）とは、△△（エンジニアが知っている概念）と同じ仕組みです。
電力系統の文脈では〜という意味で使われます。
この図解では「□□」と呼びます。」
```

### 例

| 業務用語 | エンジニア向けの説明 | たとえ |
|---------|------------------|--------|
| 潮流計算 | 電力系統全体の電気の流れを計算する処理 | ネットワーク上のトラフィック計算に相当 |
| 需給調整 | 電気の供給量と消費量をリアルタイムで釣り合わせる処理 | サーバー負荷分散に相当 |
| 系統連系 | 複数の電力系統を電気的に接続すること | ネットワーク間のルーティング設定に相当 |
| 計量値 | 電力メーターが記録した使用量データ | センサーログ・メトリクスデータに相当 |
| 整定値 | システムの動作条件を定義した設定値 | コンフィグファイルの設定パラメータに相当 |

## 変更前後比較の書き方

### 良い比較（判断できる）

```
変更前: 「需給差が〇MW以上のとき、△△処理でアラートを出す」
変更後: 「この処理を削除 → アラートが出なくなる」
変化のポイント: 需給差が大きいケースでアラートが出なくなる
```

### 悪い比較（判断できない）

```
変更前: 「△△メソッドを呼び出す」
変更後: 「△△メソッドを削除」
```

**ポイント**: コードの変更ではなく「**業務的な動作の変化**」を書く。

## 影響分析の深さ目安

| 影響レベル | 説明 | 例 |
|-----------|------|-----|
| 高（red） | 業務ルール違反・データ不整合・他機能の誤動作 | 系統制御の判断値が狂う |
| 中（orange） | 特定条件下で問題が起きる可能性 | 特定時間帯のみ影響 |
| 低（green） | 実質的な影響なし（理由を明記） | この処理は既に別の箇所で代替されている |

### 「影響なし」と言うときは理由を必ず書く

```
❌ 悪い例: 「影響なし」
✅ 良い例: 「〇〇処理が既に同じ計算を行っているため、この処理を削除しても最終結果に変化はない」
```

## 判断サポートの書き方

### ゴール: 図解を読んだエンジニアが自分で「OK / NG / 要確認」を言える状態にする

**OKの根拠の書き方**:
```
「〇〇という業務ルール（仕様書△△節）上、この処理は〜という条件が満たされない限り
発動しない。現システムでは〜のため、その条件は発生しない。→ 削除しても業務への影響なし」
```

**NGの根拠の書き方**:
```
「〇〇という業務ルールにより、△△のケースでこの処理が必ず必要。
削除すると〜という状態になり、〜という業務上の問題が発生する。」
```

**要確認の書き方**:
```
「〇〇については問題ないが、△△のエッジケース（〜という条件）については、
仕様担当者に確認が必要。」
```

## 品質チェックリスト

### 必須

- [ ] 業務用語には必ず「用語解説ボックス」がある（初出時）
- [ ] 変更前後の比較が「業務的な動作の変化」で書かれている（コード変更の列挙になっていない）
- [ ] 影響分析テーブルがある（影響先・内容・条件・レベル）
- [ ] 「影響なし」と書く場合は理由が明記されている
- [ ] 判断サポートセクションがある（OK / NG / 要確認のどれか）
- [ ] 仕様書の該当箇所が引用されている（ある場合）

### デザイン

- [ ] Lucide iconを使用（絵文字禁止）
- [ ] 変更前は赤系（`before-card`）、変更後は緑系（`after-card`）で色分け
- [ ] ヘッダーに「要確認 / 変更OK」バッジがある
- [ ] スマホでも読みやすいレスポンシブ対応（`md:grid-cols-2` など）

### 読みやすさ

- [ ] 電力系統・業務知識がゼロのエンジニアが読んで理解できるか
- [ ] コードの変更でなく「何が変わるか」が書かれているか
- [ ] 図解を読んで「OKかNGか」の意見が言えるか
