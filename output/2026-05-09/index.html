GitHub Actionsによるスケジュール型APIレポート：堅牢なサーバーレスパターンの実装

単一リポジトリでワークフロー定義とレポートロジックを管理し、cronトリガー＋手動トリガーの組み合わせで運用する設計パターン。JSONスナップショットとMarkdownサマリーを使い分けることで可読性と信頼性を両立する。

  • ▸「データ契約」を先に定義する：入力・出力・成功基準を実装前に明文化する
  • ▸機械可読なJSONスナップショットと人間可読なMarkdownサマリーを分けて保存する
  • ▸失敗モードを明示的に設計し、部分的な結果はサイレントに省略しない

この動きの意味：GitHub ActionsはCI/CDだけでなく、定期レポート配信の実行基盤として完成度が高い

「データ契約の先行定義」はこのツールの環境変数設計に対応する。BRAVE_API_KEY・GEMINI_API_KEY等のSecretsがまさに「入力の契約」。現状のrun.jsは出力（HTMLとURL）の契約が暗黙的なので、ログ出力を構造化JSONにする改善が次のステップとなりうる。

チーム横断でKPIを標準化し、全ステークホルダーが同一ソースから同一定義の数値を参照できる体制を構築する。データ取り込みと表示を分離する設計により、品質問題が下流に伝播しない仕組みを作る。

  • ▸KPIをチーム全体で標準化し「どのバージョンが正しいか」論争をなくす
  • ▸生データ取り込みと整形済みレポートを別レイヤーで管理し相互参照する
  • ▸バリデーションをワークフロー内に組み込み、エラーが無音で拡大することを防ぐ

この動きの意味：自動化の信頼性は「データの品質ゲート」をどこに置くかで決まる

このツールのスコアリング機能（1〜10点）がまさに「品質ゲート」の役割を果たしている。現状は閾値が10点のみだが、7点未満の記事を除外するルールを追加することで、低品質なHTMLが生成されるリスクを下げられる。
