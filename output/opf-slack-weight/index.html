最適潮流計算（OPF）

スラック変数の重みを修正する

α（スラック重み） 1 → 1000

現状でのバグ発生は確認されていないが、理論上「スラックを残した方が評価が良くなる」ケースが起こり得るため予防的に修正する

最適潮流計算とは

業務上の目的（まず読んでください）

電力系統では複数の発電機が同時に動いており、「どの発電機をどのくらい動かすか」を決める必要があります。 最適潮流計算（OPF: Optimal Power Flow）は、 電気の流れ（潮流）・電圧・電流の制約を守りながら、発電コストが最小になる発電量の組み合わせを求める制約付き最小化計算です。

今回の修正に関係するポイント（3行）
  • OPFの目的関数（最小化する評価式）= 燃料コスト + α × スラック変数
  • スラック変数 = 制約の違反量を数値で持つ変数。0なら制約OK、0より大きければ制約違反
  • α = スラック変数へのペナルティの重み。αが小さいとスラックを残してもスコアへの影響が小さい
用語の詳細解説（エンジニア向け）
「最適潮流計算」をエンジニア言語で言うと？

機械学習の学習最適化に近い概念。「複数の制約条件（電圧・電流の上限下限）を満たしながら、コスト関数（発電費用）を最小化する変数（発電量）を求める」制約付き最小化問題です。

「スラック変数」とは？

制約を「どれだけ違反しているか」を数値で持つ変数。
プログラムで例えると：assert(v <= max_v) の代わりに slack = max(0, v - max_v) として違反量を変数で保持する仕組み。

「目的関数」とは？

最適化ソルバーが「最小化しようとする評価式」。機械学習の損失関数（loss function）と同じ役割。 αが損失関数における「正則化係数」に相当します。

なぜ問題が起きるか（理論分析）

現在の α = 1 で起こり得るケースの具体例

これは机上検討によるリスク分析です。 現時点のプログラムでは実際には発生していませんが、α = 1 の場合、理論上以下のケースが起こり得ます。

αが小さいと、ソルバーが「制約をわずかに違反してでも燃料コストを下げた方がスコアが良い」と判断するケースが発生します。

現状: α = 1 の場合（起こり得るケース）
// 解A: 制約をわずかに違反（スラック = 0.05）、燃料コストは低め
目的関数値 = 燃料コスト(100.00) + 1 × スラック(0.05) = 100.05

// 解B: 制約クリア（スラック = 0）、燃料コストわずかに高め
目的関数値 = 燃料コスト(100.06) + 1 × スラック(0.00) = 100.06
→ ソルバーは「解A（100.05）」を選ぶ ＝ 制約違反の解が "最適解" として出力される可能性 ⚠
修正後: α = 1000 の場合
// 解A: 制約をわずかに違反（スラック = 0.05）
目的関数値 = 燃料コスト(100.00) + 1000 × スラック(0.05) = 150.00

// 解B: 制約クリア（スラック = 0）
目的関数値 = 燃料コスト(100.06) + 1000 × スラック(0.00) = 100.06
→ ソルバーは「解B（100.06）」を選ぶ ＝ 制約を満たした正しい最適解が得られる ✓

変更前後の比較

システムの動きがどう変わるか

変更前（現状: α = 1）
  • スラックを 0.01 だけ残すペナルティ = 1 × 0.01 = 0.01（非常に小さい）
  • 燃料コストの差がわずかでも、制約違反を選ぶ方が「スコア上は有利」なケースが起こり得る
  • 電圧・電流が制約範囲をわずかに超えた状態のまま「最適解」として出力される可能性がある
変更後（修正後: α = 1000）
  • スラックを 0.01 だけ残すペナルティ = 1000 × 0.01 = 10（燃料コストの差より十分大きい）
  • スラックを残すことが割に合わなくなり、制約クリアが常に優先される
  • 「制約を守りながらコスト最小」という本来の意図通りの解が得られる
変化のポイント

ソルバーの「判断優先順位」が変わります。
変更前（α=1）: 「わずかな制約違反ならコスト削減の方が得」という判断がされ得る
変更後（α=1000）: 「制約を守ることが絶対優先、その上でコストを最小化」という正しい優先順位になる

判断サポート

この修正は妥当か？対応が必要なことは何か？

修正すべき根拠
  • 「燃料コスト最小化」の本来の意味は「制約を満たした上でコストを最小化」であり、スラックが残る解（制約違反）は業務上無効
  • 現在 α = 1 では、燃料コストの差がスラックのペナルティを上回るケースが理論上発生し得る。α = 1000 にすることでこのリスクを排除できる
  • α = 1000（燃料コストの大きさに対して十分大きい値）は一般的に適切な設定であり、収束性を著しく損なう値ではない
修正後に確認すること（確認項目）
  • 既存の全テストケースが通ること
    変更後の計算結果が現在の期待値と一致するか確認する。 もし差異が出た場合は、そのケースで現状が制約違反の解を返していた可能性を確認する（差異が出る = 修正が効いた証拠の可能性）。
  • ソルバーが全ケースで正常に収束すること
    代表的なテストケースで「収束失敗・発散」が起きないことを確認する。 α = 1000 は理論上適切な範囲内だが、実際の収束挙動を念のため確認することを推奨。
